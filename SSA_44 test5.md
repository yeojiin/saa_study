### NAT GW
- 공용 서브넷은 애플리케이션 로드 밸런서를 배포하는 데 사용되고, 두 개의 사설 서브넷은 Amazon EC2 인스턴스에 애플리케이션을 배포하는 데 사용됨 개발 팀은 Amazon EC2 인스턴스가 인터넷에 액세스할 수 있기를 원하고 완전 관리형이고 IPv4를 이용함
- 퍼블릭 서브넷에 Nat GW를 구축
	- NAT GW를 사용하면 사설 서브넷의 인스턴스가 인터넷이나 다른 AWS서비스에 연결이 가능함 
	- NAT 게이트웨이에 정확히 하나의 Elastic IP 주소를 연결할 수 있음
	- NAT 게이트웨이는 TCP, UDP 및 ICMP와 같은 프로토콜을 지원
	- 보안 그룹을 NAT 게이트웨이에 연결할 수 없음
	- 네트워크 액세스 제어 목록(네트워크 ACL)을 사용하여 NAT 게이트웨이가 위치한 서브넷을 오가는 트래픽을 제어할 수 있음
	- 프라이빗 서브넷에 있는 인스턴스에 인터넷 액세스를 제공하려면 퍼블릭 서브넷에서 NAT 게이트웨이를 사용
- <u>인터넷 게이트웨이는 공용 서브넷에 배포 되어야 함</u>
- NAT Instance도 대안이 가능하지만 해당 서비스는 관리를 해야해서 올바르지 않음

	- 개인정보와 같은 민감한 정보는 S3 메타데이터에 배치하지 말 것을 권장

### Route53 Health Check ❓14

- Amazon Route 53 DNS 페일오버는 배후의 ELB와 통합하여 이러한 모든 장애 시나리오를 처리
- 활성화되면, Route 53은 개별 ELB 노드에 대한 상태 점검을 자동으로 구성하고 관리
- Amazon Route 53은 또한 ELB가 수행하는 Amazon EC2 인스턴스 상태 점검의 이점을 활용
- Amazon Route 53 DNS 페일오버는 Amazon EC2 인스턴스와 ELB의 상태 점검 결과를 결합하여 로드 밸런서의 상태와 배후의 Amazon EC2 인스턴스에서 실행 중인 애플리케이션의 상태를 평가할 수 있음
- ALB 상태 검사 사용은 LB 상태 검사를 통해 인스턴스의 지정된 TCP 포트가 연결을 수락하고 있는지 확인하거나 지정된 페이지가 200의 오류 코드를 반환했는지 확인히기 때문에 이 오류는 지정된 페일오버 시나리오에 적합하지 않음

- ALB(Application Load Balancer)를 사용하여 비디오 스트리밍 제품을 만드는데 Application Load Balancer는 정상적인 인스턴스 풀에서 인스턴스가 작동하지 않는 것으로 감지될 때마다 인스턴스를 제거하지만 Auto Scaling 그룹은 대체 인스턴스를 킥인하고 프로비저닝하지 못할 경우?
- 자동 스케일링 그룹이 Amazon EC2 기반 상태 검사를 사용하고 Application Load Balancer가 ALB 기반 상태 검사를 사용하고 있음
	- 자동 스케일링 그룹에는 자동 스케일링 및 관리를 위한 논리 그룹으로 취급되는 Amazon EC2 인스턴스 모음이 포함
- Application Load Balancer는 EC2 기반 상태 검사를 사용할 수 없음
### EBS
- EC2 인스턴스와 독립적인 영구 저장을 제공하는 Amazon Elastic Block Store(Amazon EBS) 볼륨에 업무상 중요한 데이터를 저장하려고 함,  Amazon EC2 인스턴스를 종료할 때 연결된 Amazon EBS 볼륨도 손실되었다는 것을 발견 정상적으로 작동하지 않음
- Amazon EBS 볼륨이 Amazon EC2 인스턴스의 루트 볼륨으로 구성됨, 인스턴스를 종료할 때 기본 동작은 연결된 루트 볼륨도 종료하는 것
-  Amazon Elastic Block Store(EBS)는 모든 규모의 처리량과 트랜잭션 집약적인 워크로드를 위해 Amazon Elastic Compute Cloud(EC2)와 함께 사용하도록 설계된 사용하기 쉬운 고성능 블록 스토리지 서비스
- Amazon EBS가 지원하는 AMI의 루트 볼륨은 인스턴스가 종료되면 삭제되고, 기본 동작을 변경하여 인스턴스가 종료된 후에도 볼륨이 지속되도록 할 수 있음
- 볼륨이 연결된 인스턴스를 종료한 후에도 루트가 아닌 EBS 볼륨은 계속 사용할 수 있음
